# Нормализация телефонов пользователей — дизайн

Дата: **2026-06-14**. Статус: одобрено пользователем, готово к плану реализации.

## Проблема

Вход в систему (OTP по SMS) ищет пользователя по **точному совпадению** телефона
(`auth.service.ts:41` — `findUnique({ where: { phone } })`), а DTO входа требует строго
`/^\+7\d{10}$/` (`auth/dto/request-code.dto.ts:5`, `verify-code.dto.ts:5`). При этом синк
amoCRM сохраняет телефон контакта **как есть** (`amocrm-sync.service.ts:36-39`, значение из
`amocrm-http.client.ts:31` по регексу `/^\+?\d{10,}$/`), без приведения к единому формату.

Итог: в прод-БД телефоны в разных форматах, и большинство реальных клиентов **не может войти**.

### Факты из прод-БД (2026-06-14, 546 пользователей)

| Формат | Кол-во | Войдёт сейчас? |
|---|---|---|
| `+7XXXXXXXXXX` (корректный) | 197 | ✅ да |
| `8XXXXXXXXXX` | 224 | ❌ нет |
| `7XXXXXXXXXX` (без `+`) | 98 | ❌ нет |
| `XXXXXXXXXX` (10 цифр) | 24 | ❌ нет |
| иностранные/битые | 3 | ❌ нет |

**~64% (349 из 546) реальных клиентов сейчас не могут залогиниться.** Нормализуемы 346.

**Коллизия (1 пара):** два разных контакта amoCRM с одним номером, у каждого свой заказ —
после нормализации схлопываются в один `+79167724547` (уникальный индекс на `phone`):
- `+79167724547` — `amocrm_contact_id` 27506185, 1 заказ-клиент, 0 сессий.
- `9167724547` — `amocrm_contact_id` 27509965, 1 заказ-клиент, 0 сессий.

**Неустранимые (3, у каждого реальный заказ):** `+992927077539` (Таджикистан),
`+3197010206674` (битый `+31`, 15 цифр), `+7981972536` (`+7`, но 9 цифр — обрезан).

## Решение (одобрено)

Подход А: **единый нормализатор телефона, применяемый на всех границах + одноразовая миграция**.
Цель — устранить корень навсегда (а не только починить данные, которые синк снова бы испортил).

Вход — **прощающий**: принимаем привычные форматы и нормализуем на сервере.

### 1. Ядро — чистая функция `normalizePhone(raw: string): string | null`

Единственный источник истины. Алгоритм:
1. Если `raw` пустой/`null` → `null`.
2. Срезать всё, кроме цифр и ведущего `+` (убрать пробелы, скобки, дефисы, `()`).
3. Привести к `+7XXXXXXXXXX`:
   - уже `+7` и далее 10 цифр → как есть;
   - `8` + 10 цифр → `+7` + те же 10 цифр;
   - `7` + 10 цифр (без `+`) → `+` + всё;
   - ровно 10 цифр → `+7` + всё;
   - иначе (иностранные `+992…`, битые, неверная длина) → **`null`**.
4. Результат — либо валидный `+7XXXXXXXXXX`, либо `null` (не удалось нормализовать).

Расположение: общий модуль/утилита в `apps/api/src` (напр. `common/phone.ts`), без внешних
зависимостей. Покрыта юнит-тестами на все ветки + мусорный ввод.

> YAGNI: НЕ тянем `libphonenumber-js` (данные на 99,5% российские; иностранные — вне рамок).

### 2. Применение на границах кода

| Место | Что меняем |
|---|---|
| `amocrm-http.client.ts` / `amocrm-sync.service.ts` | Нормализовать `contact.phone` перед `upsert`. Если `null` — **сохранять исходное значение как есть** (не падать; сейчас бросает `Error` при отсутствии телефона — поведение для иностранных не ломать). |
| `auth/dto/request-code.dto.ts`, `verify-code.dto.ts` | Ослабить валидацию: принимать привычные форматы (`8…`, `7…`, `+7…`, с пробелами/скобками). Жёсткий `/^\+7\d{10}$/` снять. |
| `auth.service.ts` (requestCode, verifyCode) | Нормализовать ввод **в начале**, до всех обращений к БД (`findUnique`, rate-limit, `authCode`, `upsert`). Если `normalizePhone` вернул `null` → понятная ошибка валидации («проверьте номер телефона»). |
| `admin-users.service.ts` | Нормализовать `args.phone` перед `findUnique`/`create`. |

После этого все записи и все сравнения идут по единому `+7XXXXXXXXXX`, формат ввода не важен.

### 3. Миграция существующих данных

Prisma-миграция (+ при необходимости скрипт на TypeScript/SQL), **идемпотентная, в транзакции**:
1. Нормализовать 346 записей (`8XXX`/`7XXX`/10 цифр → `+7XXXXXXXXXX`) тем же алгоритмом, что и util.
2. **Слияние коллизии** `9167724547` → `+79167724547`:
   - переназначить заказ(ы) дубля (`orders.client_user_id`, `orders.partner_user_id`) на пользователя
     `+79167724547`;
   - перенести/удалить связанные строки дубля (sessions=0, authCodes, pushTokens и т.п. — если есть);
   - удалить пользователя-дубль `9167724547`.
   - Заказы остаются привязаны по `amocrm_deal_id`; будущий синк контакта 27509965 найдёт слитого
     пользователя по нормализованному телефону и подтянет заказ к нему.
3. **3 неустранимых** (`+992927077539`, `+3197010206674`, `+7981972536`) — **не трогать**.

> Перед миграцией на проде: бэкап БД (`pg_dump`). Миграцию написать так, чтобы повторный прогон
> не ломал уже нормализованные данные (idempotent).

### 4. Поведение для иностранных/битых номеров

Останутся со своим исходным номером. Войти по `+7`-флоу не смогут (`normalizePhone` → `null` →
ошибка «проверьте номер»). Чинится правкой номера в amoCRM заказчиком или будущей поддержкой
международных номеров (вне рамок этой задачи).

### 5. Тестирование

- **Юнит** (`normalizePhone`): все форматы (`+7…`, `8…`, `7…`, 10 цифр, со скобками/пробелами),
  и `null` для иностранных/битых/пустых.
- **E2E**: `request-code` для одного пользователя в разных форматах ввода находит **одну** запись
  и возвращает 200; невалидный ввод → 400 с понятной ошибкой.
- **Миграция**: проверка идемпотентности (повторный прогон — без изменений и без ошибок);
  проверка слияния коллизии (заказ переехал, дубль удалён).

## Вне рамок (YAGNI)

- Международные номера во входе (Таджикистан и пр.).
- `libphonenumber-js` и подобные тяжёлые зависимости.
- Формат **отображения** телефона в UI (хранение `+7XXXXXXXXXX` достаточно).
- Дедупликация всех возможных будущих коллизий amoCRM (обрабатываем известную 1 пару; новые
  предотвращаются нормализацией на записи).
