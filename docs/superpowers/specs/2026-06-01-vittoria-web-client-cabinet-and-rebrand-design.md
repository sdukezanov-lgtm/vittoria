# VITTORIA HOME — веб-кабинет клиента + ре-брендинг веб-части (дизайн-спека)

Дата: 2026-06-01
Статус: согласовано в брейншторме, ожидает реализации
Автор правок: заказчик (через владельца проекта)

---

## 1. Контекст и цель

Заказчик прислал два визуальных референса нового фирменного стиля **VITTORIA HOME**
(тёплый бежево-золотой, засечный логотип, «карточный» лёгкий UI):

- **Референс 1** — главный экран **личного кабинета клиента**: шапка с логотипом и блоком
  «Сервисный отдел», карточка договора (№, изделие, статус, суммы), блок «Статус заказа»
  с процентом готовности и горизонтальным степпером из 7 этапов, блок «Быстрый доступ».
- **Референс 2** — **веб-панель** (экран «Заказы»): логотип, левое меню
  (Заказы / Чат / Партнёры / Комиссии / Аудит / Шаблоны), таблица заказов с цветными
  бейджами этапов и прогресс-барами.

Полный объём, который хочет заказчик (зафиксирован в брейншторме):

1. **Веб-кабинет клиента** — новый, в браузере, на `vittoria-home.ru`.
2. **Ре-дизайн веб-панели** (admin/партнёр) под тот же бренд, на `admin.vittoria-home.ru`.
3. **Мобильные приложения** (Android + iOS) — тот же бренд-стиль.
4. **Деплой** на `vittoria-home.ru` (+ поддомены `admin.`, `api.`).

**Эта спека покрывает только веб-часть — пункты 1 и 2.** Мобильные (п. 3) и деплой (п. 4) —
отдельные этапы со своими спеками/планами. Порядок выбран потому, что веб виден на
`localhost` в браузере и служит «эталоном» стиля для последующей сборки мобильных.

### Цель
Реализовать в браузере личный кабинет клиента по референсу 1 и перекрасить существующую
веб-панель по референсу 2, не меняя серверную часть и бизнес-логику.

## 2. Не-цели (явно НЕ делаем в этой работе)

- Не меняем backend, REST API, схему БД, бизнес-логику, интеграции (amoCRM, SMS).
- Не добавляем новых полей данных. Веб-кабинет показывает только то, что уже отдаёт
  клиентский API.
- Не делаем реальные фото изделий (в данных заказа поля фото нет) — используем
  фото-заглушку. Реальные фото — потенциальная будущая задача с доработкой backend.
- Не трогаем структуру меню и состав страниц админки — только внешний вид.
- Не занимаемся мобильными приложениями и деплоем в рамках этой спеки.

## 3. Данные, доступные клиенту (из существующего API)

Клиентский API (тот же, что использует мобильное приложение) уже предоставляет:

- **Авторизация:** `POST auth/request-code`, `auth/verify-code`, `auth/refresh`, `auth/logout`
  (вход по телефону + SMS-код).
- **Профиль:** `GET/PATCH me`, `POST me/consent`, `DELETE me`.
- **Заказы:** `GET orders`, `GET orders/{id}`, `GET orders/{id}/history`.
- **Чат:** `GET orders/{id}/chat`, `GET/POST chats/{id}/messages`, `PATCH chats/{id}/read`.
- **Сервис:** `GET service/contact` (телефон + часы сервисного отдела).

Поля заказа (`OrderDto`): `contract_number`, `product_name`, `total_amount`,
`prepayment_amount`, `balance_due`, `current_stage`, `progress_percent`, `service_phone`,
`last_admin_comment`, `partner_services[]`, `created_at`, `updated_at`.

**Чего нет в данных** (учтено в дизайне): фото изделия, подзаголовок-«коллекция»
(«современная классика»). Статус «Действующий» вычисляется на фронте из этапа/прогресса.

7 этапов (порядок и подписи фиксированы, совпадают с референсом):
`preparation_for_production` → `detailing` → `materials_arrival` → `production` →
`transfer_to_warehouse` → `completeness_check` → `ready_for_delivery`.

## 4. Бренд-токены (единая тема для всего веб-фронта)

Тема задаётся один раз и применяется и к кабинету клиента, и к админке.

- **Цвета:**
  - графит (текст, логотип «VITTORIA») — тёмный нейтральный ≈ `#2B2B2B`;
  - **золото/бронза** (акцент: «HOME», прогресс, активный этап, основные кнопки) ≈ `#B08D57`;
  - тёплый светлый беж (фон страницы) ≈ `#F6F3EE`;
  - белый (карточки) `#FFFFFF`;
  - зелёный (предоплата, статус «Действующий», 100 %) ≈ `#4F8A5B`;
  - набор приглушённых цветов для бейджей этапов (золотой / голубой / зелёный и пр.).
- **Шрифты:** засечный для логотипа и крупных заголовков (напр. Cormorant Garamond /
  Playfair Display, подключённый веб-шрифтом); гротеск для интерфейсного текста (напр. Inter).
- **Формы:** скруглённые карточки (radius ~16px), мягкие тени, увеличенные отступы.

Точные hex-коды и шрифт финализируются в коде и правятся вживую на `localhost` по
обратной связи владельца.

## 5. Кабинет клиента (новый) — страницы

Реализуется как роль `client` внутри существующего веб-приложения `apps/admin`
(переиспользуются готовый вход и API-клиент). Отдельный клиентский layout в бренд-стиле.

1. **Вход** — телефон + SMS-код (существующий механизм), оформленный в бренде.
2. **Главная** (референс 1):
   - шапка: логотип «VITTORIA HOME» + слоган, блок «Сервисный отдел» (телефон из
     `service/contact`, кликабельный);
   - карточка договора: № договора, название изделия, бейдж «Действующий»,
     **фото-заглушка**, три метрики — Стоимость / Предоплата / Остаток к оплате;
   - блок «Статус заказа»: текущий этап, крупный процент готовности, **горизонтальный
     степпер 7 этапов** (пройденные/активный — золотом, будущие — серым);
   - блок «Быстрый доступ»: карточки-ссылки на Чат, Историю этапов, Звонок в сервис.
   - если у клиента несколько заказов — переключатель заказов.
3. **История этапов** (`orders/{id}/history`) — вертикальный таймлайн с датами и комментариями.
4. **Чат с сервисом** (`orders/{id}/chat` + `messages`) — переписка, отправка сообщений и
   вложений.
5. **Профиль** (`me`) — данные пользователя, согласие, выход.

## 6. Ре-дизайн админ-панели (референс 2)

- Единая бренд-тема перекрашивает админку автоматически (фон, акценты, типографика).
- Точечно приводим к референсу: логотип «VITTORIA HOME» (засечный), цветные бейджи
  этапов в таблице, прогресс-бары с процентом.
- Состав меню, маршруты, логика и данные — без изменений.

## 7. Архитектурный подход

Расширяем существующее SPA `apps/admin` (React 18 + TypeScript + Vite + Mantine 7 +
React Query + React Router):

- добавляем роль `client`: `RoleGate allow={['client']}`, `RoleHome` редиректит клиента
  в кабинет (`/cabinet`);
- новый `ClientLayout` (бренд-шапка вместо админского AppShell) и клиентские страницы
  под `/cabinet/*`;
- общая бренд-тема Mantine (`MantineProvider theme=...`) + подключение веб-шрифтов;
- переиспользуем существующие `auth/`, `api/client.ts`, типы.

На проде один и тот же бандл обслуживает оба домена; маршрутизация по роли разводит
клиента (`vittoria-home.ru`) и сотрудников (`admin.vittoria-home.ru`). Возможное
разделение на отдельные бандлы — решение этапа деплоя, вне этой спеки.

## 8. Проверка и приёмка

- Существующие тесты `apps/admin` (vitest) остаются зелёными; для новых клиентских
  страниц добавляются базовые тесты по образцу существующих.
- Линтер/типы чисты (`tsc`, eslint).
- **Главная приёмка — визуальная:** запуск `pnpm dev:admin` + `pnpm dev:api`,
  владелец открывает `http://localhost:5173`, заходит тестовым клиентом
  (`+79991234567`, код в консоли API) и сверяет вид с референсами. Правки по цветам и
  деталям вносятся вживую.

## 9. Открытые вопросы / зависимости

- **Деплой (п. 4):** тип хостинга `vittoria-home.ru` пока неизвестен. Backend требует
  VPS с Docker (Node + PostgreSQL + Redis + хранилище). Уточняется у провайдера/заказчика
  отдельно; на дизайн веб-части не влияет.
- **Реальные фото изделий:** при необходимости — отдельная задача с доработкой backend.
