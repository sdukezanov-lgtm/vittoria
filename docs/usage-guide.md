# VITTORIA HOME — руководство по работе с приложением

Простыми словами: что это, как связано, как пользоваться, как запустить и что нужно для «боевого» режима.

---

## 1. Что это за система

Это сервис для клиентов мебельной компании. Он состоит из четырёх частей:

1. **Backend (сервер)** — «мозг» системы (NestJS). Хранит данные, общается с amoCRM, отправляет SMS и уведомления.
2. **Веб-панель** (admin + кабинет партнёра) — сайт для сотрудников. Открывается в браузере.
3. **Мобильные приложения** для клиентов — iOS и Android (код написан; собираются на Mac / в Android Studio).
4. **Интеграция с amoCRM** — `vittoriaamo.amocrm.ru`. amoCRM — источник истины по сделкам.

**Главная идея:** менеджеры ведут сделки в amoCRM как обычно, а приложение показывает клиенту этап и % готовности его заказа и даёт чат с сервисом.

---

## 2. Как всё связано (важно понять)

```
Менеджер в amoCRM двигает сделку по воронке «VITTORIA HOME»
        │  (этапы = стадии готовности заказа)
        ▼
Backend узнаёт об изменении → обновляет заказ у клиента
        ▼
Клиент в приложении видит новый этап и % + получает уведомление
```

- **Воронка в amoCRM: «VITTORIA HOME»** с 7 этапами (см. §8). Менеджер просто перетаскивает сделку на нужный этап — приложение подхватит это автоматически (этап берётся из статуса сделки в воронке).
- Дополнительные данные (% готовности, комментарий, предоплата) хранятся в служебных полях сделки `vittoria_*` (их создал я, см. §8).

## 3. Веб-панель администратора

**Адрес (локально):** http://localhost:5173
**Вход:** по номеру телефона + код из SMS (в режиме разработки код виден в консоли сервера; в боевом — приходит реальной SMS).

Разделы (левое меню):
1. **Заказы** — список всех заказов: договор, изделие, этап, % готовности. Поиск и фильтр по этапу. Клик по строке → карточка заказа.
2. **Карточка заказа** — данные сделки + редактируемые поля: этап, % готовности, комментарий. Кнопка «Сохранить» обновляет заказ (и, в боевом режиме, синхронизируется с amoCRM) и уведомляет клиента.
3. **Чат** — переписка с клиентами: слева список диалогов (с бейджем непрочитанных), справа выбранный диалог + поле ответа. Обновляется автоматически (~10 сек). При открытии диалога непрочитанные гаснут.
4. **Партнёры** — список партнёров-дизайнеров; кнопка «Создать партнёра».
5. **Комиссии** — вознаграждения партнёров: список, фильтр по статусу, создание, смена статуса (Ожидает → Одобрено → Выплачено).
6. **Аудит** — журнал действий (кто что менял), с фильтром и просмотром «было/стало».
7. **Шаблоны уведомлений** — тексты push/SMS, редактируются без перевыпуска приложения.

---

## 4. Кабинет партнёра (роль «partner»)

Тот же сайт, но партнёр входит под своей ролью и видит только своё:
1. **Мои заказы** — заказы, где он привязан как партнёр (только просмотр).
2. **Карточка заказа** — данные без редактирования.
3. **Мои вознаграждения** — его комиссии с фильтром по статусу.
4. **Профиль** — имя, телефон; редактирование имени; выход.

Партнёр **не видит** чужих клиентов, не пишет в клиентские чаты и не меняет статусы.

## 5. Мобильное приложение клиента (iOS + Android)

Экраны:
1. **Вход** — телефон → код из SMS → (один раз) согласие на обработку данных.
2. **Главный** — карточка заказа: договор, изделие, финансы (стоимость/предоплата/остаток), этап + прогресс-бар, комментарий менеджера, прогноз услуг, кнопка «Чат с сервисом», телефон сервиса (тап → звонок).
3. **История этапов** — таймлайн прохождения 7 этапов.
4. **Чат** — переписка с сервисом, отправка текста (и фото).
5. **Профиль** — имя, телефон, выход, удаление аккаунта.

> Код приложений готов в `apps/android/` и `apps/ios/`. Их сборка и публикация — см. `docs/mobile-build-and-publish.md` (нужен Mac для iOS, Android Studio для Android, аккаунты разработчика).

## 6. Как запустить локально (для демо/разработки)

Нужен установленный Docker. По шагам:

1. Поднять инфраструктуру (база, Redis, хранилище файлов):
   ```
   pnpm dev:infra
   ```
2. Применить миграции базы (один раз):
   ```
   pnpm --filter @vittoria/api prisma:migrate:deploy
   ```
3. Запустить сервер:
   ```
   pnpm dev:api
   ```
   (API поднимется на http://localhost:3000)
4. Запустить веб-панель:
   ```
   pnpm dev:admin
   ```
   (откроется на http://localhost:5173)
5. Войти: телефон администратора, затем код. В режиме разработки **код печатается в консоль сервера** (строка `[DEV-SMS] ... text="VITTORIA HOME: 1234"`).

Тестовый администратор уже заведён: телефон `+79990000000`.

## 7. Боевой режим: что включить

| Что | Как включить | Статус |
|---|---|---|
| Реальный amoCRM | Токен в `apps/api/.env` (`AMOCRM_ACCESS_TOKEN`) + `AMOCRM_CLIENT_MODE=http` | ✅ подключено |
| Реальные SMS (SMSC.ru) | В `.env`: `SMS_PROVIDER_MODE=smsc` + `SMSC_LOGIN` + `SMSC_PASSWORD` (+ `SMSC_SENDER`) | ⏳ ждём логин/пароль SMSC |
| Push-уведомления | `PUSH_PROVIDER_MODE=real` + ключи FCM (Android) и APNs (iOS) | ⏳ для мобильного релиза |
| Входящие вебхуки amoCRM | Нужен публичный адрес сервера (деплой/домен) — amoCRM не достучится до localhost | ⏳ этап деплоя |

> Секреты живут только в `apps/api/.env` (в Git не попадают). Для боевого сервера их хранят в защищённом хранилище.

## 8. amoCRM: воронка и поля (уже настроено)

**Воронка «VITTORIA HOME»** (создана, id 10959102), 7 этапов в порядке:
1. Подготовка для производства
2. Деталировка
3. Поступление материалов на склад
4. Производство изделия
5. Передача готового изделия на склад
6. Проверка комплектности товара
7. Готовность к передаче клиенту

Менеджеры двигают сделку по этим этапам — приложение показывает клиенту соответствующий этап.

**Служебные поля сделки** (созданы, только для нашего сервиса, существующие поля не трогаются):
`vittoria_stage`, `vittoria_progress`, `vittoria_admin_comment`, `vittoria_prepayment`, `vittoria_partner_user_id`, `vittoria_partner_services`.

> Если воронку/поля пересоздать — нужно обновить их ID в конфигурации (скрипт `scripts/amocrm-create-pipeline.py` помогает получить ID воронки и этапов).

## 9. Где что лежит (структура проекта)

- `apps/api/` — сервер (NestJS).
- `apps/admin/` — веб-панель (React).
- `apps/android/` — Android-приложение (Kotlin).
- `apps/ios/` — iOS-приложение (Swift).
- `docs/` — документация (этот файл, контракт API, гайд по сборке мобильных, спецификации/планы в `docs/superpowers/`).
- `scripts/` — служебные скрипты (настройка amoCRM).
- `infra/` — Docker-окружение для разработки.

## 10. Что ещё предстоит (для полного запуска)

1. **SMSC.ru** — прислать логин/пароль → включу реальные SMS.
2. **Мобильная сборка** — на Mac (iOS) и в Android Studio (Android): собрать, проверить по сценариям, опубликовать в App Store и Google Play (нужны аккаунты разработчика).
3. **Деплой сервера** — выложить backend на сервер с HTTPS-доменом; тогда заработают входящие вебхуки amoCRM (мгновенная синхронизация) и приложения будут ходить на боевой адрес.
4. **Push-ключи** — FCM (Android) и APNs (iOS) для уведомлений в мобильных.

Готов сопровождать каждый из этих шагов.
